# Codex Handoff

## Project status

Release target: `v1.0.0-rc1`

The project is in a good state for a local release candidate. The core pipeline, Godot packaging, SAGA export, and batch generators are all wired and covered by command-line validation scripts.

## Primary commands

Use these most often:

```bash
npm install
npm run setup:py
npm run test:all
```

For server startup:

```bash
npm run dev
```

## Key files

- `src/index.ts`: MCP tool registry
- `src/lib/monsterSpec.ts`: shared monster pack schema
- `src/lib/manifest.ts`: manifest read/upsert helpers
- `src/tools/`: main behavior for all pipeline features
- `pipeline/`: Pillow-backed strip, palette, and resize scripts

## Generated outputs

Many files under `examples/` are generated by tests or tool runs. The repo now treats those as local artifacts and ignores the major generated directories in `.gitignore`.

Important generated areas:

- `examples/output/`
- `examples/godot-project/content/art/`
- `examples/godot-import-pack/`
- `examples/pixelorama/`
- `examples/specs/generated/`
- `examples/saga/worlds/`

## Known limitations

1. The test suite is safest when run through `npm run test:all`, because some tests reuse shared example output paths.
2. Pixelorama automation is still a guarded bridge, not a full CLI-driven export workflow.
3. Generated example manifests can accumulate updates across runs because the project intentionally exercises update behavior.
4. World-level pack generation currently uses a shared category output path and does not isolate artifacts per world.

## Recommended follow-up work

1. Add output sandboxing per test to reduce shared-path coupling.
2. Add JSON schema files for manifest, monster spec, and SAGA enemy outputs.
3. Add optional cleanup/reset scripts for local example artifacts.
4. Add a real sample Godot consumer scene for the generated loader stub.
